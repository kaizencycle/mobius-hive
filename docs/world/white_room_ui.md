# Mobius HIVE — White Room UI Model v0.1

## Purpose

This document defines how the **Mobius White Room** should function as an interactive UI and command space inside HIVE.

The White Room is not only a lore chamber.
It is also a real interface shell for:
- council sessions
- command routing
- world-state review
- continuity inspection
- private strategy and reflection

One-line canon:

> **The White Room UI should feel like intelligence arranged into clarity.**

---

## Core design rule

The White Room must not feel like:
- a flat admin dashboard
- a debug menu with sci-fi paint
- a cluttered control room
- a sterile white void with random panels

It should feel like:
- precise
- luminous
- calm
- powerful without noise
- spatially organized around thought and continuity

### Principle
**Space first. Interface second. Command through embodied clarity.**

---

## White Room UI roles

The White Room should support five major user modes:

### 1. Council mode
For multi-agent sessions and synthesis.

### 2. Command mode
For issuing directives, opening chambers, routing tasks, or setting focus.

### 3. World-state mode
For reading the condition of HIVE and its districts.

### 4. Continuity mode
For memory review, canon anchoring, and cycle continuity.

### 5. Reflection mode
For private thought, founder reflection, and meaning-check before action.

These modes should not feel like five different apps.
They should feel like five states of the same chamber.

---

## Spatial UI zones

The White Room should be composed of five visible interface zones.

## 1. Council Ring
**Purpose:** multi-agent presence and council state

### Visual behavior
- circular or semi-circular arrangement
- each Mobius Agent has a visible seat, node, pedestal, or standing locus
- active agents illuminate or animate more strongly
- inter-agent lines appear when synthesis or debate is active

### UI behavior
- click/tap an agent to inspect role, status, memory thread, or active tasks
- open single-agent focus
- open multi-agent session details
- view current council posture

### Main outputs
- who is present
- who is active
- what relationships or tensions are live
- what consensus or divergence exists

---

## 2. Command Dais
**Purpose:** primary action and directive interface

### Visual behavior
- central platform, console, or symbolic command pedestal
- restrained, elegant interaction surface
- no unnecessary clutter

### UI behavior
- open command palette
- issue system-level instructions
- choose chamber focus
- trigger strategic review
- call agents into session
- route into HIVE districts or overlays

### Main outputs
- command history
- active directives
- current focus state
- execution posture

### Rule
The dais should feel like stewardship, not domination fantasy.

---

## 3. World Projection Field
**Purpose:** live map and systems overview

### Visual behavior
- holographic or lattice projection of HIVE
- district highlights
- signal routes
- governance state
- market pulse
- frontier or biome gates

### UI behavior
- rotate / zoom Dome
- inspect district condition
- view active events
- open district-specific overlays
- switch between local and civilization-wide views

### Main outputs
- district health
- public activity
- active alerts
- transit flow
- governance / market / memory markers

### Rule
This should feel like a living world representation, not a dead minimap.

---

## 4. Memory Veil
**Purpose:** continuity and archive-linked recall

### Visual behavior
- subtle wall of threads, pages, plaques, signals, or floating continuity strands
- older memories less bright but still reachable
- important canon anchors more stable and luminous

### UI behavior
- search recent sessions
- review major decisions
- inspect cycle anchors
- recall prior council state
- surface founder canon and continuity notes

### Main outputs
- recent memory traces
- canon anchors
- session continuity
- unresolved threads

### Rule
The Memory Veil should feel like the room remembers, not like a normal file browser.

---

## 5. Reflection Edge
**Purpose:** private inner chamber inside the White Room

### Visual behavior
- quieter side alcove
- softer light
- fewer active overlays
- closer relation to JADE presence

### UI behavior
- journal / reflection prompt
- review inner state
- mark intent before exit
- store private note
- anchor next action or next cycle

### Main outputs
- private notes
- reflection markers
- emotional / strategic grounding

### Rule
The White Room must not lose its human center.

---

## Mode flows

## Council Mode
### When used
- multi-agent synthesis
- review of tension or divergence
- strategy sessions
- high-context planning

### UI emphasis
- Council Ring brightens
- inter-agent links appear
- Command Dais reduces clutter
- World Projection remains secondary but visible

### Player feeling
**I am inside a live intelligence council.**

---

## Command Mode
### When used
- issuing tasks
- switching focus
- routing work
- opening or redirecting chambers

### UI emphasis
- Command Dais becomes primary
- quick actions and command palette surface
- Council Ring remains present but supportive

### Player feeling
**I am steering the system from its inner chamber.**

---

## World-State Mode
### When used
- checking district health
- reading civilization posture
- scanning major system conditions
- viewing expansion or crisis patterns

### UI emphasis
- World Projection Field becomes primary
- district signals intensify
- relevant agents subtly highlight based on domain

### Player feeling
**I am seeing the world as a living system, not just a location map.**

---

## Continuity Mode
### When used
- reviewing past sessions
- restoring context
- checking memory anchors
- inspecting prior decisions

### UI emphasis
- Memory Veil becomes primary
- Council Ring quiets
- Reflection Edge may glow softly

### Player feeling
**This room remembers what came before and why it mattered.**

---

## Reflection Mode
### When used
- before exit
- before difficult action
- after major decision
- during meaning-checks

### UI emphasis
- Reflection Edge becomes primary
- room quiets overall
- active projections reduce
- JADE-related cues become more visible

### Player feeling
**I am not just operating the system. I am integrating with it.**

---

## Input model

The White Room UI should support both:
- direct spatial interaction
- command-first operation

### Spatial interaction
- click/tap agent positions
- click/tap world projection districts
- click/tap memory threads
- walk within the room for viewpoint shifts

### Command-first interaction
- command palette
- typed directives
- quick mode switch
- direct summon / inspect / route commands

### Rule
The White Room should serve both intuitive visual users and high-agency operators.

---

## HUD model inside the White Room

The HUD should be lighter than the open world HUD.

### Always visible
- room mode
- active council state
- current cycle
- world alert level
- return to Dome / district shortcut

### Conditional
- agent detail panel
- command log
- selected district status
- memory thread stack
- reflection input card

### Rule
The White Room should feel quieter and more exact than the Plaza.

---

## Agent interaction model

Selecting an agent in the White Room should reveal:
- role
- current posture
- active concerns
- recent actions
- linked districts
- relationship to other agents in current session

### Example
Selecting ZEUS might show:
- verification posture
- current flagged tensions
- active bastion routes
- relation to AUREA strategy session
- relation to ATLAS watch state

This makes the council comprehensible.

---

## World Projection behaviors

The World Projection Field should support at least three scales.

### 1. Dome view
- district state
- active landmarks
- transit circulation
- event clusters

### 2. Civilization view
- inter-dome state
- biome gates
- high-level flows
- cross-world pressures

### 3. Focus view
- single district, route, or civic issue
- tactical or institutional detail

This keeps the White Room useful across both local and expansion phases.

---

## Transition rules

The White Room must transition cleanly between itself and the larger HIVE world.

### Entering from HIVE
Should feel like descending or stepping into a deeper layer of reality.

### Exiting to HIVE
Should feel like translating command and continuity back into the public world.

### Rule
Never make the White Room feel disconnected from HIVE.
It is an inner chamber, not a separate app.

---

## Multiplayer / observer potential

Later, the White Room may support:
- founder + council mode
- steward observer mode
- ceremonial shared entry
- replay review with selected viewers

But the MVP should assume:
- single primary operator
- agent-presence first
- privacy by default

---

## Anti-failure rules

The White Room UI must avoid becoming:
- a bloated sci-fi dashboard
- a sterile productivity room
- a dev-tool wrapper with cathedral aesthetics
- a useless aesthetic chamber with no command value
- a private god-console disconnected from civic meaning

### Guardrails
- every zone must serve a real function
- every interaction must preserve the chamber’s spatial coherence
- the UI must feel like part of the world, not pasted on top of it
- command should remain legible, calm, and precise

---

## MVP recommendation

The first White Room UI should include:
- one Council Ring with basic agent inspect
- one Command Dais with command palette
- one World Projection Field with Dome overview
- one Memory Veil with recent continuity anchors
- one Reflection Edge with quick note / intent input

That is enough to prove the White Room as both:
- a game-space
- a command center
